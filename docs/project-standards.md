# 项目规范

## 单元测试优先

- 新功能、缺陷修复和重构默认先补单元测试，再实现代码。
- 若已有行为缺少覆盖，先用测试锁定当前行为，再修改实现。
- 前端优先覆盖业务逻辑、数据转换和组件关键交互。
- 后端优先覆盖路由处理、参数校验、数据访问和边界条件。
- 无法补测试时，需要在提交说明中写明原因和替代验证方式。

## 代码提交规范

提交信息使用 Lora 规范，消息使用中文。

推荐格式：

```text
<类型>(<范围>): <中文描述>

<可选正文：说明为什么修改、影响范围、验证方式>
```

常用类型：

- `feat`: 新增功能
- `fix`: 修复缺陷
- `docs`: 文档变更
- `test`: 测试变更
- `refactor`: 不改变行为的重构
- `chore`: 构建、依赖、工具或仓库维护

示例：

```text
feat(api): 增加家庭成员支出查询接口

补充 SQLite 查询和路由测试，验证成员维度的支出列表返回。
```
